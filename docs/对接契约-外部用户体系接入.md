# 对接需求：用户表给问路侧的最小要求

> 场景：用户体系由同事独立开发、**另起一张用户表**。同事的代码还没上传，所以这里只列
> **问路后端这边绕不开、必须由用户表满足的最小硬性要求**。JWT / 登录方式 / 接口形态
> 等，等拿到代码后再双方对齐，本文不预设。

---

## 为什么有这份要求

问路后端切到 PostgreSQL 后，所有个人数据（大脑 / 记忆 / 器官状态 / 对话）都按
**`user_id` 隔离**，且这个 `user_id` 的类型是 **PostgreSQL `UUID`**，并外键指向用户表。

所以对接时，问路这边**只关心一件事**：能不能从你的用户表拿到一个稳定、唯一的 `UUID`
当隔离键。只要这点满足，隔离逻辑一行都不用改。

---

## 必须满足的（硬要求）

### 1. 用户表必须有一个 UUID 主键 / 唯一标识

```sql
id UUID PRIMARY KEY DEFAULT gen_random_uuid()
```

- 类型**必须是 `UUID`**（不是自增 int、不是雪花 ID、不是字符串）。
- 这一列作为用户的全局唯一身份，问路所有个人数据都靠它归属。

### 2. 这个 UUID 对同一个用户必须永远不变

- 同一个真实用户，任何时候、任何登录方式拿到的 UUID 都要**一致**。
- 一旦变了，这个用户的大脑/记忆/对话就会"换人"（查不到原来的数据）。

### 3. 用户唯一性约束至少有一个

- 用来防止同一个人被建成两行（手机号、openid、邮箱…任选，看你们业务）。
- 这条是为了保证「同一用户 → 同一 UUID」（第 2 条）能落地，具体用哪个字段你们定。

---

## 不要求 / 你们自己定

- 用户表叫什么名字、还有哪些业务字段（密码、昵称、第三方绑定、风控等）——**全你们定**。
- 登录方式、验证码、JWT 怎么签——**等代码上来再对齐**，本文不预设。
- 是否和问路放同一个库——都行，后面再说。

---

## 一个要提前知道的衔接点（不是要求，是提示）

问路现有的个人数据表外键指向的是**问路自己的 `users` 表**。两套系统合并时，需要有一步把
你的用户 UUID「认到」问路这边（最简单就是用同一个 UUID 往问路 `users(id)` 插一行）。
**这步具体怎么做，等你的代码上来后我们再定**，现在你只要保证上面 3 条硬要求即可。

另外有一个保留 UUID 不要给真实用户用：`00000000-0000-0000-0000-000000000000`
（问路迁移期的系统默认用户）。

---

## 小结（给同事一句话）

> 你的用户表加一个 `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`，保证同一个用户这个
> id 永远不变，再有个唯一约束防重复建号，就够了。其它（JWT、登录、字段）等你代码上来我们再接。
